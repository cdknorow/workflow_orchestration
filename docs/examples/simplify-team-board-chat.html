## Security Review: internal/auth/, internal/license/, internal/config/ I've completed my analysis. Here are findings organized by: (1) security vulnerabilities, (2) duplicate/simplifiable code, (3) architectural improvements. --- ### SECURITY FINDINGS (Priority) **S1. CRITICAL: middleware.RealIP before auth.Middleware causes auth bypass risk** - `server.go:178` applies `middleware.RealIP` *before* `auth.Middleware` at line 195. - `middleware.RealIP` rewrites `r.RemoteAddr` from `X-Forwarded-For` headers. - `auth.IsLocalhost()` (keystore.go:157) checks `r.RemoteAddr` to decide localhost bypass. - **Result**: An attacker can send `X-Forwarded-For: 127.0.0.1` to bypass all authentication when the server is exposed on 0.0.0.0. - **Fix**: Either (a) remove `middleware.RealIP` since auth relies on raw RemoteAddr, or (b) move `auth.Middleware` *before* `RealIP`, or (c) have `IsLocalhost` check the original remote addr via a separate mechanism. The comment on keystore.go:156 says it uses raw RemoteAddr, but RealIP middleware defeats this. **S2. HIGH: WebSocket origin checking completely disabled** - `websocket.go:36`: `InsecureSkipVerify: true` disables all WebSocket origin validation. - Combined with the SameSite=Lax cookie (which allows GET-initiated navigations), a malicious page could open a WebSocket to the Coral server and read session data. - **Fix**: Implement proper origin checking matching the CORS config instead of InsecureSkipVerify. **S3. MEDIUM: Session cookie missing Secure flag** - `keystore.go:190-198`: `SetSessionCookie` never sets `Secure: true`. - When served over HTTPS, the cookie can leak over HTTP connections. - **Fix**: Set `Secure: true` when not in dev mode, or when the request came over TLS. **S4. MEDIUM: No CSRF protection on state-changing endpoints** - `POST /auth/key`, `POST /api/system/api-key/regenerate`, `POST /api/license/activate` rely solely on SameSite=Lax cookies. - SameSite=Lax still allows top-level navigation POSTs from external sites in some browsers. - **Fix**: Consider adding a CSRF token or requiring the API key in a custom header (which triggers CORS preflight). **S5. MEDIUM: No request body size limit on license activation** - `auth/routes.go:29` correctly uses `MaxBytesReader(1MB)` for /auth/key. - `license/middleware.go:72` does NOT limit the request body for /api/license/activate. - **Fix**: Add `r.Body = http.MaxBytesReader(w, r.Body, 1<<20)` to the Activate handler. **S6. LOW: Rate limiting not applied to license activation** - Auth key validation has rate limiting (10 attempts/minute/IP). - License activation has no rate limiting, allowing brute-force attempts against the Lemon Squeezy API. - **Fix**: Add similar rate limiting to the license activation endpoint. **S7. LOW: CORS allows all origins when bound to 0.0.0.0** - `server.go:179-184`: When host is 0.0.0.0, ALL origins are allowed with credentials. - This means any website can make authenticated API requests if the user has a session cookie. - This compounds S2 (WebSocket) and S4 (CSRF). **S8. INFO: Expired sessions not cleaned from memory** - `keystore.go:124-135`: `ValidateSession` checks expiry but never deletes expired sessions. - Only LRU eviction removes sessions. If session creation rate is low, expired sessions accumulate. - Low impact since maxSessions=100, but cleaner to prune on validation. --- ### DUPLICATE CODE / SIMPLIFICATION **D1. Duplicate Routes struct pattern** - Both `auth.Routes` and `license.Routes` follow the exact same pattern: struct wrapping a manager, NewRoutes constructor, handler methods. - Not worth merging (they serve different domains), but the pattern is consistent — no action needed. --- ### ARCHITECTURAL OBSERVATIONS **A1. Config's SkipLicense→DevMode coupling is hidden** - `config.go:119-121`: When `SkipLicense=true` (build flag), it silently sets `DevMode=true`. - This means partner/demo builds also skip any other dev-mode-gated behavior. Under a distributed attack, this is a slow memory leak. - **Fix**: Periodic cleanup goroutine, or switch to a fixed-size LRU rate limiter. --- ### SUMMARY RECOMMENDATIONS (ordered by priority) 1. **Fix S1 (RealIP + auth bypass)** — This is the most critical finding.
